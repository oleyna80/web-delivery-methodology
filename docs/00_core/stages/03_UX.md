# Stage: 3 UX

## Purpose

Define information architecture, user flows, page or screen structure, content needs, states, and interaction behavior before visual design and implementation planning depend on them.

## Responsible role

UX Producer or Product/UX Analyst acting as Producer.

## Supporting roles

- Product Owner for usability direction and coverage decisions.
- Architecture Owner for technical constraints that affect flows or states.
- Content, Accessibility, Frontend, or Domain Specialists when relevant.
- Orchestrator for dependency and readiness checks.

## Required inputs

| Artifact | Required status | Why required |
| --- | --- | --- |
| Project Brief | Approved | Defines users, value, scope, and acceptance boundaries. |
| Solution Brief | Approved | Defines technical constraints that affect flows, states, and interactions. |

## Preconditions

- Product intent and architecture constraints needed for UX are approved.
- User groups, primary tasks, and success criteria are known.
- UX scope can be described without requiring visual approval.

## Allowed activities

- Define information architecture, navigation, user flows, page or screen inventory, content needs, states, and interaction behavior.
- Identify empty, loading, error, permission, and edge states when relevant.
- Record usability assumptions, open content decisions, and accessibility risks.
- Produce the UX Specification.

## Required outputs

| Artifact or result | Owner | Canonical path pattern |
| --- | --- | --- |
| UX Specification | Product/UX Owner | `docs/03_templates/ux/UX_SPECIFICATION.md` when templates exist; otherwise the approved Work Block names the path. |

## Output owner

Product/UX Owner owns the user-flow and usability meaning of the UX Specification.

## Acceptance criteria

- Pages, screens, flows, content needs, and states are explicit.
- UX decisions trace to approved product and architecture inputs.
- Interaction behavior is clear enough for UI Design and Implementation Planning.
- Accessibility and content risks are recorded.
- The Owner or delegated Product Owner can judge coverage and usability direction.

## Checks and evidence

- UX review against Project Brief and Solution Brief.
- Flow and state coverage check.
- Accessibility consideration check proportional to risk tier.
- Product Owner approval evidence.

## Approver and gate

Product Owner approves usability direction and coverage. The Orchestrator records the transition and routes defects to UX, Product Definition, or Architecture as appropriate.

## Failure routes

| Failure class | Return stage | Required action |
| --- | --- | --- |
| User need or scope is wrong | Product Definition | Revise Project Brief. |
| Technical constraint invalidates UX | Solution Architecture | Revise Solution Brief. |
| Flow, state, or content coverage is incomplete | UX | Revise UX Specification. |
| Visual design problem reveals structure issue | UX | Reopen UX before continuing UI Design. |

## Risks

- Merging UX approval with visual approval.
- Omitting error, empty, loading, or permission states.
- Designing flows that conflict with architecture constraints.
- Proceeding to UI Design with unresolved content structure.

## Stop conditions

- Product scope or architecture constraints are not approved.
- Required states or flows cannot be specified with available information.
- UX work would require changing product scope or architecture decisions.

## Next allowed stages

- UI Design after UX approval.
- Implementation Planning only when product, architecture, UX, and design inputs required by the profile are approved.
- Product Definition or Solution Architecture when the defect originates upstream.
